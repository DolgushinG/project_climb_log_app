# Backend: API отчёта об ошибках

Клиент отправляет отчёты об ошибках приложения на бэкенд, чтобы пользователь мог нажать «Отправить ошибку» при сбое.

## Endpoint

```
POST /api/app/error-report
```

**Авторизация:** опциональна. Если пользователь авторизован — передаётся Bearer token.

## Тело запроса

```json
{
  "message": "Не удалось загрузить данные",
  "screen": "checkout",
  "event_id": 123,
  "platform": "android",
  "app_version": "1.0.0+3",
  "stack_trace": "Error: ...\n  at ...",
  "extra": {}
}
```

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| message | string | да | Текст ошибки (макс. 4000) |
| screen | string | нет | Экран/страница, где произошла ошибка (checkout, group-checkout, profile, analytics и т.д.) |
| event_id | int | нет | ID соревнования (если применимо) |
| platform | string | да | android, ios, web, linux, macos, windows |
| app_version | string | нет | Версия приложения (например 1.0.0+3) |
| stack_trace | string | нет | Стек вызовов (макс. 8000) |
| extra | object | нет | Доп. контекст: `exception` (текст ошибки), `type` (тип исключения) |

**Поведение клиента:**
- При офлайне отчёт сохраняется в очередь и отправляется при появлении сети.
- Одинаковые ошибки (message+screen+event_id) не отправляются повторно в течение 24 часов.

## Ответ

**200** — успешно:
```json
{ "ok": true }
```

**401** — необязательно требовать авторизацию; можно принимать анонимные отчёты.

## Рекомендации для бэкенда

- Логировать отчёт (файл, Sentry, и т.д.)
- Ограничить частоту: например, 10 отчётов/мин на пользователя или IP
- Не возвращать клиенту детали — достаточно `{ "ok": true }`
