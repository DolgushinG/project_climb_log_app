# Backend: Ограничение учеников тренера и отзыв приглашений

Спецификация для лимита учеников (3 бесплатно, безлимит по подписке) и механизма отзыва приглашений.

---

## 1. Лимит учеников

| Условие | Лимит |
|---------|--------|
| Подписка активна или пробный период (`premium_subscriptions` / `premium_trials`) | Безлимит |
| Иначе (бесплатный доступ) | 3 слота |

**Слот** = подтверждённый ученик (в `trainer_students`) **или** отправленное pending-приглашение (в `trainer_student_invitations` со статусом `pending`).

---

## 2. Проверка лимита

### 2.1 POST /api/trainer/students (отправка приглашения)

Перед созданием записи в `trainer_student_invitations`:

1. Проверить: `(COUNT(trainer_students) + COUNT(pending invitations)) < limit`
2. Если лимит достигнут → `403 Forbidden`:

```json
{
  "error": "trainer_students_limit_reached",
  "message": "Достигнут лимит учеников. Оформите подписку для неограниченного числа учеников."
}
```

### 2.2 POST /api/trainer/invitations/{id}/accept (ученик принимает)

Перед добавлением записи в `trainer_students`:

1. Приглашение существует
2. Приглашение в статусе `pending`
3. Приглашение принадлежит тренеру
4. Лимит тренера: `(COUNT(trainer_students) + 1) <= limit` (учитываем будущего ученика)

Если приглашение отозвано или не найдено → `404 Not Found`.

Если лимит достигнут → `403 Forbidden`:

```json
{
  "error": "trainer_students_limit_reached",
  "message": "У тренера достигнут лимит учеников."
}
```

---

## 3. Отзыв приглашения тренером

### 3.1 DELETE /api/trainer/invitations/{id}

Тренер отменяет своё отправленное pending-приглашение.

| Критерий | Описание |
|----------|----------|
| Метод | DELETE |
| Права | Только тренер, владелец приглашения (`trainer_id = current_user_id`) |
| Условие | Приглашение в статусе `pending` |
| Результат | Удаление записи или перевод в `revoked` |
| Response | `200` или `204` |

**Ошибки:**
- `403` — не тренер или не владелец приглашения
- `404` — приглашение не найдено
- `422` — приглашение уже принято или отозвано

**Эффект:** слот освобождается: `(students + pending)` уменьшается на 1. Тренер может пригласить другого пользователя (в том числе тот же email).

---

## 4. Сценарии и защита от обхода

| Сценарий | Поведение |
|----------|-----------|
| Ученик принимает отозванное приглашение | `404` — приглашение не найдено/недействительно |
| Ученик принимает, когда тренер достиг лимита | `403` `trainer_students_limit_reached` |
| Тренер отзывает приглашение | Слот освобождается, можно приглашать снова |
| Цикл пригласить → отозвать → пригласить | Нормальное использование. Лимит проверяется при каждом POST. |

**Правила на бэкенде:**
1. **Accept:** проверять существование приглашения, статус `pending`, лимит тренера.
2. **Revoke:** только приглашения со статусом `pending`.
3. **POST /trainer/students:** проверка `(students + pending) < limit` перед созданием.

---

## 5. Определение лимита на бэкенде

Логика (псевдокод):

```
has_premium = EXISTS(premium_subscriptions WHERE user_id = trainer_id AND expires_at > now())
          OR EXISTS(premium_trials WHERE user_id = trainer_id AND ends_at > now())

limit = has_premium ? null : 3  // null = безлимит
```

## 6. Чеклист

- [ ] POST /api/trainer/students — проверка лимита перед созданием приглашения
- [ ] POST /api/trainer/invitations/{id}/accept — проверка приглашения и лимита перед добавлением в trainer_students
- [ ] DELETE /api/trainer/invitations/{id} — отзыв приглашения тренером (только pending)
- [ ] Accept возвращает 404 для отозванных/несуществующих приглашений
- [ ] 403 с `trainer_students_limit_reached` при превышении лимита
