# AI Chat — согласие на обработку данных (для бэкенда)

## Обзор

**Память включена по умолчанию.** Новый пользователь сразу получает персонализацию. Сообщение в UI: *«Память включена: AI запоминает уровень, цели, травмы. Изменить можно в настройках конфиденциальности.»*

Кнопка «Настройки конфиденциальности» открывает диалог для переключения:
- **«Согласен»** — переписка и факты сохраняются (значение по умолчанию)
- **«Без памяти»** — работа как «чистый лист», данные не сохраняются между сессиями

---

## Поведение клиента

### При согласии («Согласен»)
- Сохраняет и передаёт `conversation_id` при каждом запросе
- Загружает список чатов (`GET /api/ai/conversations`)
- Открывает чаты по `conversation_id`
- Сохраняет историю локально для офлайн-поддержки

### При отказе («Без памяти»)
- **Никогда не передаёт** `conversation_id` в `/chat` и `/chat/async`
- **Не вызывает** `GET /api/ai/conversations` — список чатов пуст
- При каждом «Новый чат» отправляет только `message` и `history` (пустой или из текущей сессии)
- Локально очищает сохранённый `conversation_id`, историю и pending task при смене на «Без памяти»

---

## Влияние на бэкенд

### 1. Запросы без `conversation_id`

Клиент в режиме «Без памяти» всегда отправляет запросы **без** `conversation_id`. Поведение бэкенда:
- Создаётся новый диалог при каждом первом сообщении в «сессии»
- `conversation_id` в ответе клиент **игнорирует** (не сохраняет)

Текущая логика бэкенда (создание нового диалога при отсутствии `conversation_id`) подходит.

### 2. Извлечение фактов (memories)

При отказе пользователь ожидает, что **факты о нём не сохраняются**. Варианты реализации:

**Вариант A (рекомендуется):** Сохранять предпочтение на бэкенде
- Добавить поле `ai_memory_consent: boolean` в профиль пользователя
- Клиент при выборе «Согласен»/«Без памяти» вызывает, например: `PATCH /api/users/me` с `ai_memory_consent: true | false`
- Бэкенд при `ai_memory_consent = false` **не извлекает и не сохраняет** факты в `user_memories`
- При обработке чата не подставлять `user_memories` в контекст AI

**Вариант B:** Только клиентское решение
- Бэкенд продолжает извлекать и сохранять memories при любых запросах
- Клиент просто не связывает диалоги через `conversation_id`
- Минус: факты всё равно накапливаются; при смене на «Согласен» старые memories могут «всплыть»

### 3. Хранение диалогов

Даже при «Без памяти» клиент может создавать новые диалоги (без `conversation_id`), и бэкенд их создаёт. Вопрос: нужно ли их показывать в `GET /api/ai/conversations`?

- Клиент при отказе **не вызывает** этот endpoint
- Если вызвать — логично возвращать пустой список или фильтровать по `ai_memory_consent`
- Можно оставить как есть: клиент сам не запрашивает список

## API

### PATCH /api/ai/memory-consent

Смена согласия на обработку данных. Поддерживаются Bearer и `x-telegram-init-data`.

**Body**
```json
{
  "ai_memory_consent": true
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| ai_memory_consent | boolean | `true` — сохранять переписку и извлекать факты; `false` — не сохранять memories, работать как «чистый лист» |

### GET /api/profile

В ответе есть поле `ai_memory_consent` (через `$user->toArray()`). Клиент синхронизирует состояние при смене устройства и при первом открытии экрана.

---

## Шифрование и доступ

Текст для пользователя: *«Переписка и факты хранятся в зашифрованном виде. Никто не имеет к ним доступа.»*

Бэкенд должен обеспечивать:
- Шифрование персональных данных (переписка, memories) при хранении (например, шифрование в БД)
- Отсутствие доступа к расшифрованным данным у третьих лиц и операторов
- Соответствие заявленным практикам обработки персональных данных (политика конфиденциальности, ГОСТ/ФЗ при необходимости)

---

## Сброс при выходе

При `logout` клиент очищает `ai_chat_memory_consent` из локального хранилища. Новый пользователь получит память вкл по умолчанию (без диалога).

Если предпочтение хранится на бэкенде — при смене аккаунта оно берётся из профиля нового пользователя.
